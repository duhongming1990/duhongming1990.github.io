01 / 概念

什么是时序图？

一张图让你秒懂核心本质

时序图就是一张 「谁」 在 「什么时候」 对 「谁」 说了 「什么话」 的图

⏬ 时间从上往下流动
➡️ 消息从左往右（或右往左）传递

核心目的

展示系统中不同角色/对象之间的交互顺序，让团队对"流程是怎么走的"达成共识

属于 UML 图

时序图是 UML（统一建模语言）中交互图的一种，侧重于时间维度

谁在用

后端工程师、架构师、产品经理、测试工程师，几乎所有技术角色都会接触

03 / 核心元素

时序图的 6 大组件

掌握这 6 个元素，你就能读懂和画出任何时序图

参与者 / 角色

Actor / Participant

参与交互的对象，可以是人、系统、服务、数据库等。画在图的顶部，用矩形框或小人图标表示。

生命线

Lifeline

从参与者底部向下延伸的竖直虚线，表示该对象在时间轴上的存在。时间从上到下流逝。

激活框

Activation Bar

生命线上的窄矩形条，表示对象正在处理某个任务。框的高度代表处理时长。

消息

Message

参与者之间的水平箭头，表示一次调用、请求或响应。箭头上方标注消息名称。

组合片段

Combined Fragment

用矩形框表示条件分支（alt）、循环（loop）、可选（opt）等控制逻辑。

注释

Note

附加在消息或参与者旁边的说明文字，用折角矩形表示，帮助读者理解关键逻辑。

04 / 消息类型

5 种消息箭头一览

不同的箭头样式代表不同的通信方式，这是读懂时序图的关键

➡️

同步消息

Synchronous Message

调用者发出消息后等待对方返回结果才继续执行。实心箭头表示。

⬅️

返回消息

Return Message

被调用者返回结果给调用者。用虚线箭头表示。

⚡

异步消息

Asynchronous Message

调用者发出消息后不等待返回，继续执行。开放箭头（线条箭头）表示。

🔄

自调用消息

Self Message

对象发送消息给自己，表示内部方法调用。箭头从自身出去再回来。

💥

创建/销毁消息

Create / Destroy

创建用虚线箭头指向新对象；销毁用 ✕ 标记在生命线上，表示对象被销毁。

05 / 完整示例

用户登录的时序图

一个真实场景：用户在 APP 上登录，看看各个角色是如何协作的

①

用户发起登录

用户在 APP 界面输入账号密码，点击登录按钮 → 同步消息发给前端

②

前端校验

前端先做格式校验（自调用消息），比如邮箱格式、密码长度等

③

发送 API 请求

校验通过后，前端向后端发起 POST 请求（同步消息），等待响应

④

后端安全校验

后端对参数做安全校验（自调用消息），防止 SQL 注入等攻击

⑤⑥

查询数据库

后端查询数据库验证用户信息，数据库返回查询结果（请求+返回消息）

⑦

条件分支（alt）

密码正确 → 返回 token；密码错误 → 返回 401。这就是组合片段的作用！

07 / 对比

时序图 vs 其他图

什么时候该用时序图？和其他 UML 图有什么区别？

图表类型 关注重点 适合回答的问题 视觉特征
时序图 对象之间的交互顺序 谁先找谁？消息怎么传？ 竖向时间轴 + 横向箭头
流程图 逻辑步骤的先后顺序 下一步做什么？有哪些分支？ 各种形状 + 连线
类图 类的结构与关系 有哪些类？它们什么关系？ 矩形框 + 关系线
状态图 对象状态的变化 对象有哪些状态？怎么转换？ 圆角框 + 转换箭头

💡 简单记忆：当你需要描述 「多个角色之间按时间顺序发消息」 的场景时，就用时序图。
如果只是单一流程的步骤，流程图就够了；如果关注数据结构，用类图。

08 / 实战技巧

画好时序图的 6 个技巧

来自实战经验的总结，让你的时序图更专业、更易读

🎯

1. 先定参与者

动手之前先列出所有参与者（用户、前端、后端、数据库、第三方服务），从左到右按调用频率排列。

✂️

2. 保持精简

一张图只描述一个主要流程。太复杂就拆分成多张图，不要把所有逻辑塞进一张图里。

📝

3. 消息命名要清晰

用「动词 + 名词」格式命名消息，如 queryUser()、returnToken，避免模糊的 "处理" "调用"。

🔀

4. 善用组合片段

用 alt 表示条件分支，loop 表示循环，opt 表示可选步骤，让图更完整准确。

🎨

5. 适当用颜色

不同参与者用不同颜色区分，关键消息可以加粗或高亮，帮助读者快速定位重点。

🔧

6. 用工具辅助

推荐工具：PlantUML（文本生成图）、Mermaid（Markdown 画图）、draw.io（可视化拖拽）。
