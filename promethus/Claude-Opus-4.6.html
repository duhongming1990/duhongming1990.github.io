Prometheus

核心概念 · 必知必会

Prometheus 是一套开源的系统监控与告警框架，由 SoundCloud 开发并于2016年加入 CNCF， 是继 Kubernetes 之后第二个毕业的项目。它以强大的多维数据模型、灵活的查询语言和高效的时序数据库著称。

数据模型

Data Model
📈
时间序列 Time Series
唯一标识的数据流
每条时间序列由 指标名称(metric name) 和一组 标签(labels) 唯一标识。每个时间点包含一个 float64 的值和毫秒级时间戳。
http_requests_total{method="GET", status="200"} → 1024 @1625000000
🏷
标签 Labels
多维度数据模型核心
标签是 key=value 的键值对，赋予指标多维度能力。标签分为固有标签（target自带） 和外部标签（relabel添加），任何标签变化都会产生新的时间序列。
instance job method status env region
📝
样本 Sample
时序数据的最小单元
每个样本包含三要素：完整的指标名称与标签集合、一个 float64 数值、 一个 int64 毫秒级 Unix 时间戳。Prometheus 按固定间隔采集样本。
metric + labels float64 value timestamp(ms)

四大指标类型

Metric Types
Counter 计数器
只增不减 · 累计值
📈
单调递增的累加器，值只能增加或在重启时归零。适用于请求数、错误数、任务完成数等场景。 通常搭配 rate() 或 increase() 使用。
http_requests_total{method="POST"}
Gauge 仪表盘
可增可减 · 瞬时值
🌡
可以任意上下波动的数值，代表某个瞬时状态。适用于温度、内存使用率、当前并发连接数、 队列长度等场景。
node_memory_MemAvailable_bytes
Histogram 直方图
分桶统计 · 分位数计算
📊
将观测值分配到预定义的桶(bucket)中，自动生成 _bucket、 _sum、_count 三组时间序列。 适合请求延迟、响应大小的分布分析。
histogram_quantile(0.95, rate(http_duration_bucket[5m]))
Summary 摘要
客户端计算 · 精确分位数
📋
在客户端直接计算流式 φ-分位数。生成 {quantile="0.5"}、 _sum、_count。 无法聚合，适合单实例精确分位数需求。
go_gc_duration_seconds{quantile="0.99"}
对比维度 Histogram Summary
分位数计算 服务端 (PromQL) 客户端 (应用内)
可否聚合 ✅ 可以跨实例聚合 ❌ 不可聚合
精确度 近似值（取决于桶边界） 精确值
性能开销 客户端低，服务端查询时计算 客户端高（流式计算）
推荐场景 SLA / SLO / 多实例聚合 单实例精确分位数

PromQL 查询语言

Prometheus Query Language
1️⃣
瞬时向量 Instant Vector
同一时刻的一组时间序列，每条只有一个样本值。最常用的类型。
http_requests_total{job="api"}
2️⃣
范围向量 Range Vector
一段时间范围内的样本集合，用 [duration] 选择，不能直接绘图。
http_requests_total{job="api"}[5m]
3️⃣
标量 & 字符串
标量(Scalar)为浮点数值，字符串(String)目前较少使用，用于表达式中的常量运算。
count(up == 1) → scalar: 5
🔢 聚合操作 Aggregation
按 job 统计请求速率 Top5
topk(5, sum by(job) (rate(http_requests_total[5m])))
统计各状态码的请求比例
sum(rate(http_requests_total[5m])) by (status)
统计在线实例数
count(up == 1)
⏱ 速率计算 Rate & Increase
每秒请求速率 (Counter必备)
rate(http_requests_total[5m])
过去1小时增长量
increase(http_requests_total[1h])
瞬时增长率 (适合波动大的指标)
irate(http_requests_total[5m])
📐 数学与过滤 Math & Filter
内存使用率 (%)
(1 - node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) * 100
CPU使用率 > 80% 的实例
100 - (avg by(instance)(irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
📊 直方图分位数 Histogram
P99 请求延迟
histogram_quantile(0.99, sum(rate(http_duration_seconds_bucket[5m])) by (le))
平均请求延迟
rate(http_duration_seconds_sum[5m]) / rate(http_duration_seconds_count[5m])

服务发现 & 核心组件

Service Discovery & Components
🎯
服务发现
Service Discovery
自动发现监控目标，无需手动配置每个实例。支持多种发现机制，动态适配云原生环境中服务的弹性伸缩。
kubernetes_sd consul_sd ec2_sd dns_sd file_sd static_config
📦
Exporter 导出器
将指标转化为Prometheus格式
第三方系统的指标桥接器，将非Prometheus格式的指标转换为标准格式暴露在 /metrics 端点上。
node_exporter mysqld_exporter redis_exporter blackbox_exporter cadvisor
📤
Pushgateway
短生命周期任务的桥梁
为无法被拉取的短期任务（如 CronJob、批处理脚本）提供 Push 模式。任务将指标推送到 Pushgateway， Prometheus 再从其拉取。注意：不应滥用为通用 Push 网关。
batch job cronjob 短生命周期
🔔
Alertmanager
告警管理中心
接收 Prometheus 推送的告警，负责去重、分组、抑制、静默和路由，将告警通知发送至邮件、 Slack、Webhook、PagerDuty 等渠道。
分组 Grouping 抑制 Inhibition 静默 Silence 路由 Routing
💾
TSDB 存储引擎
Local Time Series Database
Prometheus 自研的高性能本地时序数据库，按 2 小时 Block 组织数据，使用 WAL 保障写入安全。 支持 Remote Write/Read 对接远端长期存储。
Block WAL Compaction Remote Write
🏷
Relabeling 标签重写
采集前后的标签魔法
在采集前(relabel_configs)和采集后 (metric_relabel_configs)对标签进行修改、过滤、映射， 是高级配置的核心能力。
replace keep drop hashmod labelmap

规则引擎

Recording Rules & Alert Rules
Recording Rule 
# 预计算规则 — 加速复杂查询
groups:
  - name: http_rules
    rules:
      - record: job:http_requests:rate5m
        expr: sum by(job)(
          rate(http_requests_total[5m])
        )

      - record: job:http_latency:p99
        expr: histogram_quantile(0.99,
          sum by(job, le)(
            rate(http_duration_bucket[5m])
          )
        )
Alert Rule 
# 告警规则 — 触发条件 + 持续时间
groups:
  - name: instance_alerts
    rules:
      - alert: HighMemoryUsage
        expr: (
          1 - node_memory_MemAvailable_bytes
              / node_memory_MemTotal_bytes
        ) * 100 > 85
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "内存使用率超过85%"
          description: "{{ $labels.instance }} 内存 {{ $value | printf \"%.1f\" }}%"

global:
  scrape_interval:     15s       # 全局采集间隔
  evaluation_interval: 15s       # 规则评估间隔
  scrape_timeout:      10s       # 采集超时时间

    - static_configs:
        - targets: ['alertmanager:9093']

  - "rules/recording_rules.yml"
  - "rules/alert_rules.yml"

scrape_configs:
  # 监控 Prometheus 自身
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  # 主机监控
  - job_name: 'node'
    file_sd_configs:
      - files: ['targets/nodes.json']
        refresh_interval: 30s

  # Kubernetes Pod 自动发现
  - job_name: 'kubernetes-pods'
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: true

🔀
分组 Grouping
将相同性质的告警合并为一组通知。例如集群大面积故障时，按 cluster 和 alertname 分组，避免告警风暴。
🚫
抑制 Inhibition
当某条高优先级告警触发时，自动抑制相关的低优先级告警。例如节点宕机告警触发后， 抑制该节点上的所有服务告警。
🔇
静默 Silence
在指定时间窗口内，对匹配的告警进行静默处理。适合计划维护、已知问题期间临时屏蔽告警通知。

最佳实践

Best Practices
01

指标命名规范

使用 库名_子系统_单位_total 格式，如 http_server_requests_seconds_total，保持统一后缀（_total / _bytes / _seconds）。

02

控制标签基数

避免高基数标签（如 user_id、request_id）。标签值的排列组合直接决定时间序列数量， 基数爆炸会导致 OOM。

03

Counter 必用 rate()

永远不要直接对 Counter 画图。使用 rate() 或 increase() 转换为速率，自动处理重启归零问题。

04

善用 Recording Rules

对频繁使用的复杂查询预计算结果，降低 Dashboard 加载时的查询压力，显著提升 Grafana 响应速度。

05

Histogram 优先于 Summary

在多实例环境下优先使用 Histogram，因为它支持服务端聚合。只有在需要精确分位数且单实例场景下才选 Summary。

06

告警设置 for 持续时间

避免瞬时波动触发告警。建议 for: 5m 以上，根据业务容忍度调整， 减少告警噪音。

07

长期存储方案

本地存储仅适合短期。使用 Remote Write 对接 Thanos / VictoriaMetrics / Cortex 实现长期存储和全局视图。

08

监控 Prometheus 自身

关注 prometheus_tsdb_head_series（活跃序列数）、 采集延迟、WAL 大小等内部指标，防止监控系统本身成为盲区。
